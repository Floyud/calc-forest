# 开发与应用报告

案例名称：我的计算森林：面向小学数学错因诊断与引导的教育智能体

## 一、开发背景

小学数学课堂中，教师每天面对大量计算作业，但真正耗时的不是判断对错，而是判断学生为什么错、下一步该怎样讲、怎样在有限时间内准备合适的巩固练习。

在真实教学中，主要有三个痛点。第一，教师难以区分错因。同一道题的错误可能来自退位处理不稳定、进位遗漏、运算顺序错误或基础事实不熟，只看最终答案很难决定该讲"方法"还是讲"习惯"。第二，短时、分层、教材对齐的反馈准备成本高。课堂小练通常只有三到五分钟，教师既要对齐教学进度，又要兼顾不同层次学生。第三，学生在错题场景中常常只得到"错了"或"答案是什么"，缺少温和的学习反馈。

2025 年教育部明确提出十三岁以下未成年人不得独立使用生成式人工智能，教师需要一种既能利用 AI 提升效率、又能保持教学主导权的工具。基于这些需求，我们借助 Dify 智能体平台和生成式大模型，自主开发了"我的计算森林"——一个面向小学数学教师的错因诊断与引导教育智能体。

## 二、设计与开发

### （一）平台与技术选择

本项目采用 Dify 智能体平台加 FastAPI 规则服务的混合架构，核心设计原则是"算术判断归规则，语言表达归大模型"。

Dify 是国产智能体搭建平台，负责工作流编排、知识库检索、提示词管理和结果组织。我们设计了三个核心工作流：教师诊断助手、学生引导助手和 AI 批改画像助手，分别对应错因诊断、分层引导和批量批改三个教学场景。FastAPI 负责结构化数学错因诊断和练习推荐，之所以单独搭建规则服务，是因为算术正确性必须由确定性引擎保证，不能交给概率性的大模型。前端采用 Next.js，数据库使用 SQLite，大模型接入 DeepSeek 和智谱 GLM 等国产模型并支持三级回退。

### （二）知识库建设

知识库是本系统的教学专业性基础。我们借助大模型辅助梳理了人教版一到六年级数学教材的计算题型，构建了一百四十六个结构化知识库文档，组织为七大领域。

错因分类体系是知识库的核心，共四十四种错因类型，分为两个维度。知识类错因二十三种（编码 E-K01 到 E-K23），指向概念理解偏差，如乘法口诀错误、忽略括号优先级、分数加减分母也变等。习惯类错因二十一种（编码 E-H01 到 E-H21），指向计算过程疏漏，如进位遗漏、退位遗漏、数位未对齐、数字抄写错误等。每个错因文档采用 YAML 元数据头，包含错因编码、关联年级、跨年级出现场景和关联知识点编号，支持工作流中的语义召回和交叉检索。

年级知识点文档覆盖一到六年级上下册，按"年级加学期加单元加知识点"四层编号（如 G4-UP-03-01 表示四年级上册第三单元第一节），每个文档包含计算法则、典型例题、常见错误及关联错因编码，与错因分类体系形成双向链接。教学策略文档定义了四步引导法（接纳与安抚、算理引导、自主纠正、巩固确认）和三种引导模式（标准、探索、挑战），为大模型生成引导话术提供专业框架。此外还包括批改理念、出题原则、难度分级标准和六年级上册五个单元三级难度的题库。

知识库通过脚本同步到 Dify 平台的向量知识库中，支持工作流运行时的语义检索。这种"本地 Markdown 为源、Dify 向量库为用"的设计，保证了知识库可版本管理、可独立更新、可被工作流和规则引擎同时引用。

### （三）Dify 工作流设计

我们设计了三个 Dify 工作流，采用"HTTP 请求调用规则服务、Code 节点负责数据转换、If-Else 节点实现条件路由"的多节点编排架构。

以教师诊断助手工作流为例，其流程为：接收教师输入（题目、标准答案、学生作答、年级），经 Code 节点规范化为标准 JSON 后，调用 FastAPI 的 POST /api/diagnose 接口获取结构化诊断结果。随后通过 If-Else 节点按置信度路由：置信度大于等于零点六五时进入高置信度路径，依次调用练习推荐接口（POST /api/practice/recommend）、可选读取树种和鼓励语配置，最终组装为教师可读、学生温和的结构化输出；置信度不足时进入保守路径，返回"先追问、先补步骤"的草案而非强行诊断。

学生引导助手工作流在此基础上增加了 Dify 内部的知识库检索节点，从向量知识库中召回该错因对应的教学策略和教材方法，由大模型生成四步引导话术。AI 批改画像助手工作流则面向批量场景，接收多条作答记录，逐条调用诊断接口并汇总错因统计，生成班级错因画像供教师查阅。

每个工作流均保留完整的运行日志，教师可回溯每次诊断的推理过程，便于教学反思。

### （四）规则诊断引擎

规则诊断引擎是本系统保证数学准确性的核心。我们用 Python 编写了覆盖 E01 到 E11 共十一种核心错因的检测链，每种错因对应一个独立的检测函数。开发过程中大量使用大模型辅助编写正则表达式和算术验证逻辑。以下是退位错误检测的核心代码：

```python
def _detect_borrow(record, expression, computed):
    if _needs_borrow(a, b):
        no_borrow = _subtract_without_borrow(a, b)
        if computed.student == Fraction(no_borrow, 1):
            return _tag(ErrorCode.BORROW, 0.92,
                f"{a}-{b} 需要退位，学生答案符合未退位逐位相减的结果 {no_borrow}。",
                "用数位表复盘退位过程，先练 3-5 道同结构题。",
                "看到本位不够减时，先向前一位借 1，再继续算。")
```

该函数通过模拟"不退位逐位相减"来还原学生的错误思路，输出包含错因码、置信度、证据描述、教师处理建议和学生引导话术。检测链按优先级依次执行：先检查抄题转写错误，再检查括号和运算顺序，然后检查进位退位、数位对齐等计算过程错误，最后检查基础事实和概念错误。主错因取置信度最高的结果，次错因取前两项供教师参考。

### （五）功能架构

当前成果包含五个核心功能模块。错因诊断：教师输入题目、标准答案和学生作答，系统返回主错因码、置信度、证据和教师建议，覆盖十一种核心错因。AI 辅助批改：支持批量作业批改，AI 自动诊断后教师逐条审核确认。分层引导：按标准、探索、挑战三种模式生成四步引导话术，永远不直接给答案。短时练习推荐：围绕当前错因自动生成三到五分钟巩固题，支持三级难度。教师审核门：所有 AI 输出默认待审核状态，教师可逐条采纳、修改或驳回。

## 三、应用过程与效果

本项目适用于小学数学课堂小练、作业讲评、错题订正和单元复习场景。

以典型演示案例为例，教师输入四年级题目"402 减 178"、标准答案 224、学生作答 334。系统返回主错因 E03 退位错误，置信度百分之九十二，证据为"402 减 178 需要退位，学生答案符合未退位逐位相减的结果 334"。同时生成教师摘要"学生在含零的连续退位减法中退位处理不稳定"和三道同类巩固练习题，供教师审核后使用。

我们通过合成数据模拟器验证了系统的覆盖度和稳定性。模拟器按人教版六年级教学进度，生成八周、三档学生水平（优秀、中等、薄弱）的作答数据，覆盖 E01 到 E11 全部错因类型。模拟器的核心设计是：错误答案不是随机生成的，而是通过与规则引擎相同的算术模拟逻辑（如"不退位逐位相减"）计算得出，确保模拟数据与诊断引擎的检测规则严格匹配。配套资源中保留了核心测试记录、演示数据和可复现运行说明，便于评审和其他教师复核。

教学效益体现在三个方面。第一，提高讲评效率。教师不必逐题人工归因，可先查看结构化诊断草案快速决定讲评重点。第二，增强针对性。从"按题讲答案"转向"按错因讲方法"，同一错因的学生可归为一组集中讲评。第三，保护学习节奏。学生侧强调温和引导，不排名，用森林成长隐喻提供正向激励。

## 四、创新与反思

本案例的创新主要体现在四个方面。第一，从"答对答错"推进到"错因加审核加引导"的教师工作流，实现十一种细颗粒度错因分类，每种错因配有结构化证据和教师处理建议。第二，规则引擎与大模型的混合架构，算术判断由规则引擎完成保证准确可解释，引导话术由大模型生成保证个性化，两者职责清晰分离。第三，教师主导的审核门机制，所有 AI 输出默认待教师审核，符合未成年人保护和教师专业判断边界。第四，基于 YAML 元数据的知识库交叉链接体系，错因、知识点、教学策略和题库之间可双向检索，为大模型生成提供结构化的教学专业上下文。

当前不足：规则诊断对更复杂应用题的覆盖有限，低年级口算规则还需补强，真实教学场景的效果数据仍需积累。后续改进方向：补强低年级题型规则，完善教师审核反馈闭环，把采纳、修改、驳回的结果反哺知识库和提示词优化。

## 合规说明

本案例坚持教师主导、人机协同、数据脱敏和未成年人保护原则。演示材料统一使用合成样例数据，不展示真实学生姓名和正面画面。算术对错判断仅由规则引擎完成，大模型不参与答案正确性判定。AI 生成的错因建议、引导话术和练习草案均需教师审核后使用。如材料中出现生成式人工智能生成的文本、图片或语音，均按要求标注"AI 生成"。
