# 技术栈与竞品格局

## 项目定位

本项目面向“小学数学错因诊断教育 Agent”，核心目标不是替学生直接给答案，而是基于题目、学生作答过程和最终答案，识别错误类型、追问关键步骤、生成个性化讲解与巩固练习。技术路线采用 **Dify-first、Coze-ready**：优先用 Dify 交付可演示、可评测、可迭代的主工作流，同时保留 Coze 迁移与备选实现能力。

## 推荐技术栈

| 层级 | 首选方案 | 备选方案 | 说明 |
| --- | --- | --- | --- |
| Agent 编排 | Dify Workflow | Coze Bot / Workflow | Dify 负责主链路、节点可视化、日志与迭代；Coze 用于平台备选与展示冗余。 |
| LLM 接入 | Dify Model Provider | Coze 模型配置 / 自建 OpenAI-compatible 网关 | 统一通过平台配置模型，减少代码侧耦合。 |
| 业务 API | FastAPI | Flask | 提供题目解析、错因标签规范化、评测集运行、结果导出。 |
| Python 环境 | `/home/lyzhang/miniconda3/envs/pyt0` | Docker 镜像封装 | 本地开发、离线评测、脚本运行均使用该 Conda 环境。 |
| 数据存储 | PostgreSQL / SQLite | Supabase / MySQL | Demo 阶段可 SQLite，评测与多用户演示建议 PostgreSQL。 |
| 向量检索 | Dify Knowledge / pgvector | Milvus / Chroma | 存放错因知识库、教学策略、题型规则与示例。 |

## Dify-first 的原因

1. 工作流节点清晰，适合把“题目理解、答案校验、错因诊断、追问、讲解、练习推荐”拆成可观察步骤。
2. 内置知识库、变量、条件分支和 HTTP 调用，能快速连接自建诊断 API。
3. 便于竞赛演示：评委可以看到可视化链路、日志、节点输出和迭代空间。
4. 能把模型能力与业务规则分离：模型负责推理和表达，API 负责结构化校验与标签规范。

## Coze-ready 的原因

1. Coze 对 Bot 形态、插件调用和多轮会话展示友好，可作为备用演示入口。
2. 竞赛或部署环境如果限制 Dify，可将核心 prompt、工具接口和知识库迁移到 Coze。
3. Coze 可用于对比实验：同一批题目和错因标签，在不同编排平台上跑一致性评测。

## 核心能力地图

| 能力 | 实现方式 | 产物 |
| --- | --- | --- |
| 题目解析 | LLM 节点 + 规则校验 API | 题干、年级、知识点、标准解法草案 |
| 学生答案理解 | LLM 抽取步骤，必要时 OCR 后处理 | 学生最终答案、关键中间步骤 |
| 错因诊断 | 规则标签 + LLM 判断 + 置信度 | 错因标签、证据句、诊断理由 |
| 苏格拉底追问 | 基于错因类型选择追问模板 | 1-3 个引导问题 |
| 个性化讲解 | 分层提示，不直接替代思考 | 概念解释、纠错步骤、易错提醒 |
| 巩固练习 | 按知识点、错因和难度生成 | 2-3 道变式题与答案要点 |
| 安全约束 | Prompt 规则 + 输出检查 | 不羞辱、不贴标签、不越级教学 |

## 技术风险

1. **错因误判**：LLM 可能用合理但错误的解释填补证据。需要结构化证据字段和人工抽检。
2. **直接给答案**：教育场景要求先引导再讲解。需要在 prompt 和评测中检查“是否过早暴露完整答案”。
3. **年级适配不足**：同一知识点对三年级和六年级解释深度不同。数据模型必须保存年级、教材版本和先修知识。
4. **题目输入噪声**：手写、OCR、口述题目可能不完整。工作流应先做澄清，而不是强行解题。
5. **平台迁移成本**：Dify 与 Coze 的节点能力不同。应保持 prompt、API schema、错因 taxonomy 平台无关。

## 建议里程碑

1. Demo 版：文本输入题目与学生答案，输出错因诊断、引导问题和巩固题。
2. 增强版：接入 Dify Knowledge，支持按错因检索教学策略。
3. 评测版：用 JSONL 测例批量调用 API 与 Dify workflow，生成诊断准确率报告。
4. 备用版：将核心 prompt 和 HTTP 工具迁移到 Coze，保留相同输出 JSON schema。
5. 展示版：准备 3 个典型错因案例和 2 分钟演示视频脚本。
