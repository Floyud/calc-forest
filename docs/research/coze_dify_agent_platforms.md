# Coze / Dify 智能体平台调研

## 结论

本项目建议采用 **Dify-first, Coze-ready**：

- **Dify-first**：第一版以 Dify 作为教育 Agent 的工作流底座，便于自托管、复现、记录日志、接入自研 API，并能在《创AI》开发报告中清楚展示“教师借助生成式 AI 自主开发”的证据链。
- **Coze-ready**：保留 Coze/扣子作为展示入口或第二阶段产品壳。Coze 的智能体体验更接近真实用户能感知的“教育 Agent”，适合演示和快速验证教师对话流程。
- **核心能力不锁死在平台内**：错因诊断、分层练习、学生画像更新、题目校验必须沉淀在自研 FastAPI 服务中，由 Dify/Coze 通过 HTTP 工具调用。这样既能换平台，也能保证数学判断可测试、可解释、可复现。

## 平台定位对比

| 维度 | Dify | Coze / 扣子 | Coze Studio |
| --- | --- | --- | --- |
| 适合场景 | 自托管 AI 应用、RAG、Workflow、Agent、API 集成 | 快速搭建面向用户的智能体体验 | 开源智能体开发平台，适合研究和私有化探索 |
| 对本项目价值 | 适合做可复现的比赛工程底座 | 适合做教师端 Agent 演示入口 | 可作为 Coze 生态的本地化备选 |
| 优势 | 工作流、知识库、日志、API、模型管理较完整 | 对话体验和智能体产品感强，低代码配置快 | 开源、可研究平台实现 |
| 风险 | 需要我们自己做更好的前端展示 | 云端平台涉及学生数据脱敏和合规边界 | 本地部署和维护成本高于直接使用云平台 |
| 建议角色 | 第一版主平台 | 第二阶段展示壳 | 备选研究方向 |

## 为什么不能只做平台智能体

平台智能体擅长编排流程，但小学数学计算诊断不能完全交给大模型：

- 大模型可能把错误解释得很像，但数学判断不一定稳定。
- 同一道题需要区分“算错”“抄题错”“步骤漏写”“进位/退位错”，必须有结构化规则和证据。
- 比赛要求作品资料完整、便于复现；只给平台截图和 prompt 不够有工程说服力。
- 学生数据涉及未成年人隐私，不应把原始作业图片、姓名、班级等直接交给外部模型。

因此平台负责“对话、工作流、知识库、工具调用”，自研服务负责“数学可靠性”。

## Dify 工作流适配方式

第一版 Dify 应用建议做成 Workflow/Agent 混合：

1. 教师输入班级、学生编号、题目、学生答案、可选步骤。
2. Dify 做基础字段校验和脱敏提醒。
3. Dify 通过 HTTP 节点调用 `/api/diagnose`。
4. Dify 根据诊断结果调用 `/api/profile/update`。
5. Dify 调用 `/api/practice/recommend` 生成分层练习草案。
6. Dify 用 LLM 生成教师可读摘要，但不修改诊断标签。
7. 输出时明确标注“待教师审核”。

## Coze 展示入口适配方式

Coze/扣子用于第二阶段展示时，只承担“教师助理”的外壳：

- 配置角色：小学数学错因诊断助教。
- 配置知识库：课程标准摘要、错因标签说明、双减作业原则。
- 配置插件/工作流：调用同一套 FastAPI 接口。
- 输出风格：短句、证据优先、每个建议都说明对应错因。

Coze 的对话体验可以用于演示视频中“教师向 Agent 提问”的片段，例如：

- “今天三年级 2 班竖式减法错因最多的是什么？”
- “给 S012 生成 10 分钟以内的退位减法补救练习。”
- “帮我准备明天 5 分钟课堂讲评。”

## 比赛表达建议

在《创AI》材料中，不把作品描述成“用了某个平台搭了个机器人”，而要描述为：

> 教师借助生成式人工智能平台完成教育 Agent 的交互与工作流编排，同时自主开发数学错因诊断服务和学生画像数据层，形成可复现、可推广、教师可控的小学数学作业诊断系统。

这个表述同时覆盖“AI 赋能开发”“有效解决问题”“操作简单易用”“开源分享/可复现”“真实落地”等评分点。
