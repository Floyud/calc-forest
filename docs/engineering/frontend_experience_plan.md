# 前端展示界面规划

> 目标：为 **我的计算森林** 设计一个适合小学课堂审美、同时能清晰展示项目能力边界的网页端产品壳。  
> 当前定位：**演示优先、教师端优先、保留学生视角预览**，不假装已经做成完整学生 App。  
> 补充说明：本文档是体验规划与设计参考，不替代当前比赛主口径；当前对外统一以“教师工作台 + 错因诊断闭环 + 教师审核边界 + Dify 主入口”为准。

## 一、规划结论

前端建议做成一个 **Web 单页应用（SPA）**，但信息结构上分成两种视角：

1. **教师演示主视角**
   - 让评委、老师、合作方一眼看懂：题目怎么录入、AI 怎么诊断、教师怎么审核、学生会看到什么样的引导。
   - 这是当前最真实、也最能承接现有后端能力的展示方式。

2. **学生/品牌语气预览视角**
   - 用轻量页面展示成长语气、单个学生状态、鼓励语和错题引导节奏。
   - 强调“这是产品体验方向或品牌表达层”，不与当前 MVP 的实现边界冲突。

换句话说，前端不是只做“一个好看的首页”，而是做一个 **可讲述、可演示、可继续接 API 的产品外壳**。

## 二、推荐技术栈

### 核心选择

- **Next.js 15 + React 19 + TypeScript**
- **Tailwind CSS 4**
- **shadcn/ui**
- **Framer Motion**
- **Recharts**
- **TanStack Query**
- **Zustand**

### 为什么这样选

#### 1. Next.js

适合这个项目的原因：

- 社区大，资料多，稳定度高。
- 既能快速做展示页，也能后续扩展成正式产品端。
- 对路由、静态页面、组件组织、后续部署都比较顺手。
- 如果以后要接登录、教师后台、学生页，延展性比纯 Vite 更稳。

#### 2. React + TypeScript

- 主流程度高，后面找资料、找模板、找组件都容易。
- 对组件化页面、状态拆分、交互演示友好。
- 适合把“诊断结果卡片”“品牌语气卡片”“成长时间轴”“教师审核面板”拆成独立模块。

#### 3. Tailwind + shadcn/ui

- 迭代速度快，适合先把演示壳搭出来。
- `shadcn/ui` 不是重型设计系统，方便做定制，不容易做成千篇一律后台风。
- 可以在保留工程效率的同时，把视觉调到更适合小学生课堂的方向。

#### 4. Framer Motion

- 非常适合“树木成长”“卡片浮现”“步骤推进”这种轻动画。
- 演示时能明显提升“产品感”，但不需要写很重的动画系统。

#### 5. Recharts

- 足够主流，也比很多复杂图表库更轻。
- 做班级错因分布、练习时间、成长里程碑进度够用。

## 三、不建议现在采用的路线

- **Vue + Element Plus**
  - 稳，但容易过早长成“管理后台味”，不太像小学课堂产品。

- **纯 HTML/CSS 原型**
  - 起步快，但后续接真实 API、做状态流和演示交互时会很快吃力。

- **小程序优先**
  - 现在会分散精力，不利于比赛演示与桌面浏览器展示。

- **重图形引擎 / Canvas-first**
  - 容易为了炫而炫，开发成本高，也和当前 MVP 核心价值不匹配。

## 四、界面气质方向

## 关键词

- 温暖
- 清新
- 有陪伴感
- 有成长感
- 不幼稚
- 不像培训机构打卡系统

## 视觉原则

1. **自然系配色，而不是糖果色堆砌**
   - 主色建议：森林绿、嫩芽绿、暖米白、阳光黄、果实橙
   - 辅色建议：天空蓝、木质棕
   - 避免高饱和大红大紫

2. **插画感要轻，不走幼儿园路线**
   - 对象是小学课堂，不是学龄前启蒙
   - 可以用“卡片 + 柔和图形 + 小树成长组件”，少用大面积卡通贴图

3. **教师端要专业，学生预览要有温度**
   - 教师端更强调清晰、可信、结构化
   - 学生预览更强调陪伴、鼓励、节奏感

4. **功能信息优先于装饰**
   - 漂亮可以加分，但“看懂项目功能”才是第一任务

## 五、推荐的信息架构

建议首版网页分成 6 个主页面，其中后两页默认作为可选展示层，不作为当前 MVP 主证明点。

## 1. 首页 / 项目总览

用途：

- 一屏解释“我的计算森林是什么”
- 让人快速理解“诊断 + 审核 + 引导”这条当前主能力链

建议模块：

- 顶部品牌区：标题、副标题、核心理念
- 中段能力总览：诊断、教师审核、四步引导、品牌语气
- 底部快速入口：进入教师演示 / 学生预览 / 技术架构

## 2. 教师诊断演示页

这是最重要的页面。

用途：

- 真实调用当前或模拟当前 API
- 展示教师如何录入一条作答记录
- 直接返回错因、证据、教师摘要、练习草案

页面分区建议：

- 左侧：题目录入表单
- 中间：诊断结果卡片
- 右侧：教师审核与建议动作

核心展示字段：

- 年级
- 题目
- 正确答案
- 学生答案
- 主要错因
- 证据
- 教师摘要
- 引导模式
- 待审核状态

## 3. 学生引导预览页

用途：

- 让人看见“错了以后不是直接被告知答案，而是一步步引导”
- 把四步引导法做成清楚、友好的可视化流程

页面结构建议：

- Step 1 接纳与安抚
- Step 2 教材方法引导
- Step 3 总结方法
- Step 4 巩固练习

这里可以做成“对话卡片 + 算理小提示 + 同类题”的组合。

## 4. 品牌语气页（可选）

用途：

- 讲清项目的品牌表达与低压力成长语气

建议模块：

- 单个学生的当前状态卡
- 成长语气原则说明
- 低压力、无排名、教师可控的反馈示例
- 可选展示里程碑时间轴，但不作为主卖点

核心表达：

- 不强调排名
- 不强调连续打卡
- 不把森林累计当作当前 MVP 主叙事

## 5. 树种与语气配置页（可选）

用途：

- 展示树种与鼓励语如何作为品牌表达层存在
- 说明它们不是当前教师端主流程的必需功能

建议模块：

- 树种卡片墙
- 树种详情弹窗
- 科属、习性、寓意、小知识

## 6. 技术与流程页

用途：

- 给老师、评委、合作方看清项目不是概念图，而是有实际架构

建议模块：

- 前端页面层
- Dify 工作流层
- FastAPI 服务层
- 数据配置层
- 当前已实现接口清单

## 六、首版页面优先级

### P0 必做

1. 首页
2. 教师诊断演示页
3. 学生引导预览页
4. 技术与流程页

### P1 推荐

1. 品牌语气页（可选）
2. 树种与语气配置页（可选）

### P2 后续

1. 班级画像看板
2. 假期模式预览
3. 多模态输入预览

## 八、前后端衔接策略

## 第一阶段：演示友好模式

页面可以先用两种数据来源并存：

1. **真实 API**
   - `/api/diagnose`
   - `/api/practice/recommend`
   - `/api/tree-species`
   - `/api/encouragements`
   - `/api/dify/session-draft`

2. **前端本地 mock**
   - 用于尚未实现的成长语气扩展、学生端视图、班级看板

这样可以避免前端被后端尚未完成的能力卡住。

## 第二阶段：逐步替换 mock

- 森林页接未来 `GET /api/forest/{student_id}`
- 班级看板接未来摘要接口
- 学生成长记录接未来画像服务

## 九、适合课堂与演示的交互细节

1. 表单默认带一条示例题，打开页面就能演示。
2. 诊断结果用“标签 + 证据 + 建议动作”三段式展示，避免纯大段文字。
3. 学生引导页每一步一次只呈现一个重点，避免信息太满。
4. 树木成长动画要轻，控制在 300ms 到 800ms 的过渡区间。
5. 鼓励语要像老师和伙伴在说话，不要像游戏弹奖励。
6. 在显眼位置标注“教师审核后再给学生”，保护可信度。

## 十一、首版实现顺序

1. 初始化 `Next.js + TypeScript + Tailwind + shadcn/ui`
2. 先做视觉基线：首页 + 全局样式 + 颜色/字体变量
3. 接 `/api/dify/session-draft` 做教师诊断演示页
4. 用 mock 数据做学生引导预览页
5. 补技术与流程页，服务比赛答辩和展示
6. 视时间补品牌语气页或树种/鼓励语配置页

## 十二、当前技术空白

这些点目前可以先按默认方案推进，但后面可能需要你拍板：

1. **网页主要是给谁演示**
   - 比赛评委优先
   - 一线老师优先
   - 校领导/合作方优先

2. **是否需要做双模式入口**
   - 教师端主界面
   - 学生端预览界面

3. **是否要准备大屏演示版**
   - 如果比赛现场投屏多，首页和诊断页需要做更强的大屏适配

4. **是否使用定制插画**
   - 如果你后面想把品牌感做得更强，可以再补一套树木与季节插画资源

## 十三、我的建议

如果你的目标是“既符合小学课堂审美，又能让我一眼 get 到项目功能”，那最好的首版不是传统后台，而是：

**一个带自然成长气质的教师演示产品壳，核心突出诊断闭环，学生成长感作为第二层体验。**

这样做的好处是：

- 不会偏离当前 Dify-first 教师端 MVP
- 能把“我的计算森林”最有辨识度的品牌感做出来
- 后续不管接真实 API、补比赛截图，还是扩成正式产品，都不会推翻重来
